1. **FleetOverview SSE handler rewrite** -- Replace the full-reload `LoadDataAndRefresh()` calls in the SSE event handler (lines ~107-126) with in-place patching:

   ```csharp
   await InvokeAsync(async () =>
   {
       if (evt.Kind == SessionEventKind.StateChanged)
       {
           var idx = _sessions.FindIndex(s => s.SessionId == evt.SessionId);
           if (idx >= 0)
           {
               // In-place patch: parse new state from event meta
               if (evt.Meta?.TryGetValue("newState", out var stateStr) == true
                   && Enum.TryParse<SessionState>(stateStr, out var newState))
               {
                   _sessions[idx] = _sessions[idx] with { State = newState };
               }
               else
               {
                   // Fallback: if meta doesn't have newState, fetch this session only
                   var updated = await Api.GetSessionAsync(evt.SessionId);
                   if (updated is not null) _sessions[idx] = updated;
               }
           }
           else
           {
               // Unknown session -- single API call to fetch and add
               var newSession = await Api.GetSessionAsync(evt.SessionId);
               if (newSession is not null) _sessions.Add(newSession);
           }
           StateHasChanged();
       }
       else if (evt.Kind == SessionEventKind.HostMetrics)
       {
           // Deliver host metrics to sidebar
           if (evt.Meta is not null)
           {
               OnHostMetricsReceived(evt.Meta);
           }
       }
       else if (evt.Kind == SessionEventKind.SessionCompleted)
       {
           // Update completed session status in-place
           var idx = _sessions.FindIndex(s => s.SessionId == evt.SessionId);
           if (idx >= 0)
           {
               var updated = await Api.GetSessionAsync(evt.SessionId);
               if (updated is not null) _sessions[idx] = updated;
               StateHasChanged();
           }
       }
   });
   ```

2. **Add host metrics delivery mechanism** in FleetOverview.razor:
   - Add a field: `private List<HostRecord> _hosts = [];` (may already exist for sidebar)
   - Add method:
     ```csharp
     private void OnHostMetricsReceived(Dictionary<string, string> meta)
     {
         if (!meta.TryGetValue("hostId", out var hostId)) return;
         var idx = _hosts.FindIndex(h => h.HostId == hostId);
         if (idx < 0) return;

         double? cpu = meta.TryGetValue("cpu", out var cpuStr) && double.TryParse(cpuStr, out var cpuVal) ? cpuVal : null;
         long? memUsed = meta.TryGetValue("memUsedMb", out var muStr) && long.TryParse(muStr, out var muVal) ? muVal : null;
         long? memTotal = meta.TryGetValue("memTotalMb", out var mtStr) && long.TryParse(mtStr, out var mtVal) ? mtVal : null;

         _hosts[idx] = _hosts[idx] with { CpuPercent = cpu, MemUsedMb = memUsed, MemTotalMb = memTotal };
         StateHasChanged();
     }
     ```
   - Ensure `_hosts` is passed to `HostSidebar` component (it may already be via a Hosts parameter).

3. **Update HostSidebar** to display stale data indicator per CONTEXT.md decisions:
   - HostSidebar already renders CPU/memory bars from HostRecord. Since HostRecord now has populated values (from Plan 02 EntityMappers fix and from SSE HostMetrics events), the bars will show real data.
   - The stale indicator ("--" when never collected, dimmed when > 60s) relies on MetricsUpdatedUtc which is on HostEntity, not HostRecord. For SSE-delivered metrics, they're always fresh. For polling fallback, the data from GET /api/hosts comes from DB where MetricsUpdatedUtc is set. Since HostRecord doesn't carry MetricsUpdatedUtc, add a simple heuristic: if CpuPercent is null, show "--". The dimming (data older than 60s) is less critical for v1 since SSE keeps data fresh when connected. Note this in the summary as a simplification.
   - If HostSidebar doesn't already receive hosts as a parameter, check the existing component binding and ensure the `_hosts` list from FleetOverview flows through.

1. **Fix CS8602 in SseStreamReader.cs** at line 83:
   The warning is on `sseItem.EventId.ToString()` in the nullable analysis flow. The `sseItem` variable comes from `SseParser.Create(stream).EnumerateAsync(ct)` which yields `SseItem<string>` -- a struct, so it's never null. The CS8602 is likely a false positive from the nullable analyzer not tracking the struct guarantee through the async enumerable.

   Fix by using a local variable with explicit null suppression or by restructuring:
   ```csharp
   // Option A: Suppress the specific warning (cleanest for false positive)
   var eventId = sseItem.EventId.ToString()!;

   // Option B: Add explicit check (defensive)
   var eventId = sseItem.EventId.Length > 0 ? sseItem.EventId.ToString() : "";
   ```
   Use Option B (defensive) -- it's more explicit and avoids suppression comments.
