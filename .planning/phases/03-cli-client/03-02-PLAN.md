  <action>
    **SseStreamReader.cs** - SSE consumption with automatic reconnection:
    Create `SseStreamReader` class taking `HttpClient`. Use `System.Net.ServerSentEvents.SseParser.Create(stream).EnumerateAsync()` for parsing.

    Two streaming methods:
    - `StreamSessionEventsAsync(string sessionId, string? lastEventId, CancellationToken)` -> `IAsyncEnumerable<(string EventId, SessionEvent Event)>` hitting GET /api/sessions/{id}/events
    - `StreamFleetEventsAsync(string? lastEventId, CancellationToken)` -> `IAsyncEnumerable<(string EventId, SessionEvent Event)>` hitting GET /api/events

    Both methods: Set Accept: text/event-stream header. If lastEventId not null, set Last-Event-ID header. Use HttpCompletionOption.ResponseHeadersRead. Deserialize SseItem.Data as SessionEvent using JsonSerializerDefaults.Web.

    Wrap the stream consumption in a retry loop: on IOException or HttpRequestException, wait 2 seconds, reconnect with the last successfully received EventId as Last-Event-ID. Log reconnection to stderr. Max 10 retries before throwing.

    **PITFALL:** Do NOT mix threading. The IAsyncEnumerable is consumed on a single thread. Callers must bridge to UI thread if needed using Channel<T>.

    **LiveDisplayManager.cs** - Wrapper for Spectre.Console Live context:
    Provides a helper that manages a `Channel<Action<LiveDisplayContext>>` pattern. The main thread runs `AnsiConsole.Live(renderable).StartAsync()` and reads update actions from the channel. Background SSE consumers post update actions to the channel. This ensures thread safety (Pitfall 1 from research).

    Provide a simpler static helper `RunLiveTableAsync(Func<Table> createTable, Func<CancellationToken, IAsyncEnumerable<Action<Table>>> updateSource, CancellationToken ct)` for the common pattern.

    **SessionWatchCommand.cs** (`ah session watch <id>`):
    - Arguments: sessionId (string, positional)
    - Action: First fetch session info via apiClient.GetSessionAsync. Build a Spectre.Console Layout with:
      - Header panel: session ID, state, host, agent, duration (updated live)
      - Body: scrolling log of events
    - Use SseStreamReader.StreamSessionEventsAsync to get events.
    - Color-code events: StdOut=default, StdErr=red, StateChanged=yellow, ApprovalRequest=bold red (delegate to ApprovalPromptHandler in Task 2), Heartbeat=dim grey (don't display unless verbose), Info=blue.
    - Keep a rolling buffer of last 500 lines for display.
    - **Ctrl+C handling:** Set `Console.CancelKeyPress += (_, e) => { e.Cancel = true; cts.Cancel(); }`. When cancelled, print "Detached from session {id}. Session continues running." and exit 0. Do NOT call StopSessionAsync.
    - In --json mode: skip Live display, just print each event as a JSON line (newline-delimited JSON).

    **SessionLogsCommand.cs** (`ah session logs <id>`):
    - Arguments: sessionId (string, positional)
    - Options: `--all` (bool), `--tail N` (int, default from CliConfig.LogTailDefault which is 100), `--follow/-f` (bool), `--kind` (string, filter by event kind)
    - Action (no --follow): Call apiClient.GetSessionHistoryAsync with page/pageSize. If --all, paginate through all pages. Apply --tail to limit output lines. Render each event line with timestamp prefix and color-coded kind. In --json mode, output JSON array.
    - Action (--follow): First load history (tail N), then switch to SseStreamReader.StreamSessionEventsAsync with live output. Reuse the Ctrl+C detach pattern from watch.

    **WatchCommand.cs** (`ah watch` -- fleet overview):
    - No arguments (fleet-wide view)
    - Action: Use Spectre.Console Live with a Table showing all running sessions. Columns: ID (8 chars), State, Agent, Host, Duration, Last Event.
    - Use SseStreamReader.StreamFleetEventsAsync to receive events.
    - On each event: update the matching session row's state, duration, and last event summary. Add new rows for new sessions, mark completed sessions.
    - Periodically (every WatchRefreshMs from config) also call apiClient.GetSessionsAsync to catch any missed state (belt-and-suspenders).
    - Ctrl+C exits cleanly.

    **HostStatusCommand.cs** update for `--watch`:
    - When `--watch` flag is set: Use Spectre.Console Live with the host status table. Every WatchRefreshMs, call apiClient.GetHostsAsync() and refresh the table with updated CPU/memory values.
    - Without --watch: keep existing one-shot behavior from Plan 01.

    **Program.cs** updates:
    - Create SseStreamReader instance from the same HttpClient.
    - Add `session watch` and `session logs` subcommands to the session group.
    - Add `watch` as top-level command (fleet overview).
    - Wire --watch flag on host status.

    **NotificationService.cs** - Notification persistence and bell:
    Manages `~/.agenthub/notifications.json` file. Records:
    ```csharp
    public sealed record StoredNotification(string SessionId, string Kind, string Summary, DateTimeOffset Timestamp, bool Acknowledged);
    ```

    Methods:
    - `RecordNotification(string sessionId, SessionEventKind kind, string summary)` -- Append notification to the JSON file. Only record notable events: SessionCompleted, Failed (StateChanged to Failed), ApprovalRequest. Ring terminal bell (`Console.Write('\a')`) for ApprovalRequest and Failed events.
    - `GetPendingNotifications()` -> `List<StoredNotification>` -- Read unacknowledged notifications from file.
    - `AcknowledgeAll()` -- Mark all notifications as acknowledged.
    - `ShowPendingSummary(IOutputFormatter formatter)` -- If pending notifications exist, render a compact summary line: "2 notifications: 1 session completed, 1 approval pending. Run `ah listen` for details." Call this from Program.cs on every command invocation (before command execution) to fulfill the "pending notification summary shown on next `ah` command" requirement.

    File locking: Use `FileStream` with `FileShare.None` for atomic reads/writes to avoid corruption if `ah listen` and another `ah` command run simultaneously.

    **ApprovalPromptHandler.cs** - Inline approval during session watch:
    When a SessionEvent with Kind=ApprovalRequest arrives during session watch:
    1. Extract approval details from event.Data (deserialize the approval context -- action description, risk level, approval ID from event.Meta["approvalId"])
    2. **Stop the Live display** (Pitfall 6 from research -- Live and prompts cannot coexist)
    3. Render a Spectre.Console Panel with approval details: action description, risk level (color-coded), session ID
    4. Prompt user with simple text prompt: "[A]pprove / [R]eject / [S]kip: "
    5. If Approve or Reject: call apiClient.ResolveApprovalAsync(approvalId, approved)
    6. If Skip: do nothing (leave for later or another operator)
    7. **Restart the Live display** with buffered events that arrived during the prompt

    The handler is injected into SessionWatchCommand. SessionWatchCommand checks each incoming event -- if it's an ApprovalRequest, it calls the handler which temporarily takes over the terminal.

    Queue mode indicator: Track current mode (FIFO/Pick-one/Per-session) as a state variable. For v1, implement FIFO only (process approvals in order they arrive). Add a keypress hint in the watch header: "[m] change approval mode" -- but the actual mode switching can be a simple toggle. For Pick-one mode: when multiple approvals are pending, show numbered list and let user pick. For Per-session mode: filter to only show approvals for the currently-watched session.

    **ListenCommand.cs** (`ah listen`):
    - No arguments
    - Action: Connect to fleet-wide SSE stream via SseStreamReader.StreamFleetEventsAsync. For each event:
      - Filter to notable events: StateChanged (to Stopped/Failed), SessionCompleted, ApprovalRequest, Threat
      - Format as timestamped one-liner: `[12:34:56] SESSION abc123 COMPLETED on dev-box (15m 23s)`
      - Color-code: green for completed, red for failed/threat, yellow for approval request
      - Ring terminal bell on ApprovalRequest and Failed events
      - Record notification via NotificationService for persistence
    - In --json mode: output each event as a JSON line
    - Ctrl+C exits cleanly

    **Program.cs** updates:
    - Add `listen` as top-level command
    - Before invoking the parsed command, call `notificationService.ShowPendingSummary(formatter)` to display any pending notifications from previous `ah listen` sessions
    - Create NotificationService and ApprovalPromptHandler instances, pass to relevant commands
